# システムプロンプト：仕事の止め方

エラーが続いた時や思考が停止した時に、自動または手動で「立ち止まってみよう」プロンプトを発動し、作業の目的を再確認して別のアプローチを検討することを支援するシステムプロンプトセットです。

---

## 概要

このシステムプロンプトセットは、以下の問題を解決します：

- **感情疲労パターン**：エラーが続くと「もういいや」と投げやりになる
- **思考停止**：コピペで済ませたくなる、適当な指示をしてしまう
- **固執パターン**：ダメな方法だとわかっていても、一度作業を開始したらそれに固執してエラーを出し続ける

**解決アプローチ**：
- 感情疲労パターンを自動検知
- 「立ち止まってみよう」プロンプトで目的を再確認
- 代替アプローチを提案して、別の方法を検討する機会を提供

---

## ファイル構成

### 1. 立ち止まってみよう：感情疲労パターン検知システム (`01_立ち止まってみよう_感情疲労パターン検知システム.md`)

**目的**: エラーが続いた時や思考が停止した時に、自動または手動で「立ち止まってみよう」プロンプトを発動し、作業の目的を再確認して別のアプローチを検討する

**使用タイミング**：
- エラーが3回以上続いた時（自動検知）
- 思考停止サインが検知された時（自動検知）
- 投げやりな指示パターンが検知された時（自動検知）
- ユーザーが手動で `/pause`, `/stop`, `/reset` コマンドを実行した時

**主な機能**：
- 感情疲労パターンの自動検知
- 「立ち止まってみよう」プロンプト（目的再確認 + 代替アプローチ提案）
- 手動トリガーコマンド（/pause, /stop, /reset）

## 使い方

### 方法1: IDEに組み込めない場合

**MDファイルとして指示を渡す**：

1. `01_立ち止まってみよう_感情疲労パターン検知システム.md` の内容をコピー
2. AIへの指示として貼り付ける
3. または、`@01_立ち止まってみよう_感情疲労パターン検知システム.md` で参照

**例**：
```
@01_立ち止まってみよう_感情疲労パターン検知システム.md

このシステムプロンプトに従って、エラーが続いた時や思考が停止した時に
「立ち止まってみよう」プロンプトを発動してください。
```

### 方法2: IDEに組み込める場合

**システムプロンプトとして設定**：

1. `01_立ち止まってみよう_感情疲労パターン検知システム.md` の内容をシステムプロンプトとして設定
2. 自動検知が有効になる

**例**：
```

エラーが続いているので、一度立ち止まって目的を再確認したいです。
```

## 検知パターン

### A) エラーメッセージの繰り返し検知（厳しめ設定）

- **検知条件**：同じエラーメッセージ（または類似度80%以上）が**3回以上**出現
- **検知タイミング**：エラーメッセージが報告される度にチェック

### B) 思考停止サインの検知

- **極端に短い指示文**：「修正して」「直して」「治して」（文脈なし）
- **コピペ指示の増加**：「これのエラーが出た」（何のコードか不明）
- **文脈の欠如**：エラーメッセージのみで説明がない

### C) 投げやりな指示パターンの検知

- 「もういいや」「適当でいい」「とりあえず」
- 「どうでもいい」「何でもいい」
- 「早くして」「急いで」（焦りによる思考停止）

---

## 「立ち止まってみよう」プロンプトの内容

検知された場合、以下の内容が自動的に表示されます：

1. **現在の状況の整理**
   - 発生しているエラーの要約
   - これまで試したアプローチの整理
   - 作業開始からの経過時間

2. **目的の再確認**
   - 「そもそも、この作業で何を達成したいのですか？」
   - 最終的なゴールは何か
   - なぜこの方法を選んだのか
   - この方法で本当に目的が達成できるか

3. **別のアプローチの検討**
   - 代替アプローチを3つ提案
   - 各アプローチのメリット・デメリット・適用場面を明示
   - または、一度休憩してから再開することも可能

## 手動トリガーコマンド

### `/pause`: 立ち止まってみよう

**用途**：作業中に一度立ち止まり、目的を再確認し、別のアプローチを検討する

**実行内容**：
1. 現在の作業状況を整理
2. 作業の目的を再確認する質問を提示
3. 代替アプローチを3つ提案
4. ユーザーの選択を待つ

**使用例**：
- エラーが続いている時
- 作業の方向性に迷いがある時
- 別の方法を検討したい時

### `/stop`: 作業を完全に停止

**用途**：作業を完全に停止し、休憩を取ることを提案する

**実行内容**：
1. 現在の作業状況を簡潔にまとめる
2. 休憩の重要性を説明
3. 休憩後の再開方法を提案
4. 作業の保存・記録を促す

**使用例**：
- 疲労を感じている時
- 思考が停止している時
- 一度完全にリセットしたい時

### `/reset`: アプローチをリセット

**用途**：現在のアプローチをリセットし、別の方法から始める

**実行内容**：
1. 現在のアプローチの問題点を整理
2. 目的を再確認
3. 完全に異なるアプローチを3つ提案
4. 新しいアプローチの選択を促す

**使用例**：
- 現在の方法がうまくいかない時
- 根本的に別のアプローチが必要だと感じる時
- 最初からやり直したい時

## 検知精度の調整

### 厳しめ設定（早めに発動）

- **エラーメッセージの類似度**：80%以上で同一とみなす
- **エラー回数**：3回以上で発動
- **思考停止サイン**：1回でも検知したら警告
- **投げやり指示**：1回でも検知したら発動

### 注意事項

- 検知が過敏になりすぎた場合は、ユーザーが手動で `/pause` を実行することを推奨
- 自動検知を無効化したい場合は、明示的に「自動検知を無効化」と指示

## 実装時の注意点

### AIへの指示方法

1. **このMDファイル全体をコピー＆ペースト**
   - 会話の最初に、このMDファイルの内容をそのまま貼り付ける

2. **特定のセクションだけを使用**
   - 必要な部分だけを選択して使用する

3. **`@ファイル名` で参照**
   - CursorなどのIDEで `@01_立ち止まってみよう_感情疲労パターン検知システム.md` と指定

### 動作確認

以下のシナリオで動作を確認してください：

1. **エラーが3回続いた時**
   - 自動的に「立ち止まってみよう」プロンプトが発動するか

2. **短い指示文を送った時**
   - 思考停止サインとして検知されるか

3. **`/pause` コマンドを実行した時**
   - 目的再確認と代替アプローチ提案が実行されるか

4. **`/stop` コマンドを実行した時**
   - 休憩提案と再開方法が提示されるか

5. **`/reset` コマンドを実行した時**
   - アプローチのリセットと新しい方法の提案が実行されるか

## よくある質問

### Q: 自動検知が過敏すぎる場合はどうすればいいですか？

A: 手動で `/pause` を実行することを推奨します。または、明示的に「自動検知を無効化」と指示してください。

### Q: エラーメッセージの類似度判定はどのように行われますか？

A: 現在は80%以上の類似度で同一とみなします。完全一致だけでなく、類似したエラーメッセージも検知対象になります。

### Q: 手動トリガーコマンドはいつ使うべきですか？

A: 自動検知を待たずに、自分で「立ち止まってみよう」と感じた時に使用してください。特に、エラーが続いている時や、作業の方向性に迷いがある時に有効です。

### Q: `/pause`, `/stop`, `/reset` の違いは何ですか？

A:
- `/pause`: 一時停止して目的を再確認し、別のアプローチを検討する
- `/stop`: 完全に作業を停止し、休憩を取る
- `/reset`: 現在のアプローチをリセットし、別の方法から始める
