## フェーズ 3: 仮説検証の効率化

### 目的

仮説を体系的に検証し、最短で根本原因を特定する。

### 実施内容

#### 3.1 検証計画の策定

仮説リストを基に、効率的な検証順序を決定してください。

**検証順序の決定基準**:

1. **優先度の高い仮説から**: 発生確率が高い仮説を先に検証
2. **検証の容易さ**: 簡単に検証できる仮説から
3. **依存関係**: 他の仮説の検証に影響する仮説を先に

**検証方法の選択**:

- **ログ確認**: アプリケーションログ、システムログ、エラーログ
- **コード確認**: 関連するコードのレビュー
- **設定確認**: 環境変数、設定ファイル、インフラ設定
- **検証コード作成**: 小さな検証コードで動作確認
- **環境差分比較**: 正常動作環境と異常環境の差分確認

#### 3.2 段階的検証の実施

各仮説を順次検証し、結果を記録してください。

**検証結果の記録形式**:

```markdown
### 仮説 1 の検証結果

**検証方法**: [ログ確認 / コード確認 / 設定確認 / 検証コード作成]

**検証内容**:

- [具体的な検証手順]

**検証結果**:

- ✅ **確認できたこと**: [確認できた事実]
- ❌ **確認できなかったこと**: [確認できなかったこと]
- 🔍 **新たな発見**: [検証中に発見した新しい情報]

**結論**:

- ✅ **原因として確定**: [確定した場合]
- ❌ **原因ではない**: [否定された場合]
- 🔄 **追加検証が必要**: [追加検証が必要な場合]
```

#### 3.3 検証結果の統合

検証結果を統合し、根本原因を特定してください。

**根本原因の特定基準**:

- 症状を完全に説明できる
- 検証結果で裏付けられている
- 環境差分を説明できる（環境間で挙動が異なる場合）

**根本原因が特定できない場合**:

- 追加の仮説を立てる
- より詳細な検証を実施
- ログやエラーメッセージを再確認

#### 3.4 原因特定ドキュメントの作成

以下のテンプレートを使用して、原因を記録してください：

```markdown
## 調査結果

### 根本原因の特定

**根本原因**: [具体的な原因の説明]

**根拠**:

1. [検証結果 1]
2. [検証結果 2]
3. [検証結果 3]

**症状との関連**:

- [症状 1] → [原因による説明]
- [症状 2] → [原因による説明]

**環境差分の説明**:

- [環境 1 での挙動] → [原因による説明]
- [環境 2 での挙動] → [原因による説明]

### 検証プロセスの記録

[各仮説の検証結果を記録]
```

### 確認事項

原因を特定したら、以下を確認してください：

- [ ] 根本原因が明確に特定されている
- [ ] 根拠が記録されている
- [ ] 症状との関連が説明されている
- [ ] 環境差分が説明されている（環境間で挙動が異なる場合）

**確認後、ユーザーに提示**: 「この根本原因で問題ありませんか？追加の検証が必要だと思いますか？」
